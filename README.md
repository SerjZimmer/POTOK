POTOK — руководство по текущему функционалу (Frontend + Backend)

Этот документ предназначен для заказчиков и тестировщиков. В нём описано, что уже реализовано, как запускать приложение и как проверять ключевые сценарии. Проект состоит из двух модулей: «Заметки» (Go backend + Flutter UI) и «Календарь» (Flutter, офлайн‑первый; REST для будущей синхронизации).

Быстрый старт

- Требования: Flutter 3.x, Dart, Go 1.22+ (локально), macOS/Windows/Linux. Для Android/iOS — стандартные SDK.
- Запуск backend (порт 8080):
  - make run-backend
  - Проверка: curl http://localhost:8080/healthz (200 OK)
- Запуск frontend:
  - macOS: cd frontend && make run
  - Альтернатива: cd frontend && flutter run -d chrome (Web) или flutter run -d macos
- Важно для Android‑эмулятора: backend доступен по http://10.0.2.2:8080. Сейчас фронт настроен на http://localhost:8080, поэтому рекомендуем запускать на desktop/web. При необходимости можно вывести baseUrl в конфиг по платформе.

Структура

- backend/ — Go‑сервер заметок (и точка подключения календарного API под /v1/*).
- frontend/ — Flutter‑приложение с модулями «Заметки» и «Календарь».
- openapi/calendar.yaml — OpenAPI 3.1 спецификация для будущего REST‑синка календаря.

Модуль «Заметки» (готово к использованию)

- Список папок слева: название + счётчик заметок.
- Список заметок справа: предпросмотр, тап для редактирования.
- Создание/редактирование заметок.
- Удаление свайпом влево.
- Сервер: Go/SQLite на http://localhost:8080. БД по умолчанию — backend/potok.db; включены WAL и busy_timeout для устойчивости к конкуренции.

Тестовый сценарий
1) Запустите backend: make run-backend.
2) Во фронтенде создайте пару папок и заметок, проверьте счётчики.
3) Удалите заметку свайпом — запись должна исчезнуть, счётчики обновиться.

Модуль «Календарь» (офлайн‑первый, MVP)

Календарь полностью локальный: данные сохраняются в JSON‑файл ~/.potok_calendar.json (или calendar_store.json рядом с приложением, если переменная HOME недоступна). Позже будет синхронизация через REST по OpenAPI.

Представления

- Месяц
  - Сетка 6×7, неделя начинается с понедельника.
  - Включены «хвосты» предыдущего/следующего месяцев — их события видны сразу.
  - Плашки событий (до 3 на день) + индикатор +N при переполнении.
  - Навигация по месяцам стрелками в AppBar. Сегодня подсвечен.
  - Долгое нажатие по дню — быстрое создание события в этот день.
  - Короткий тап — переход на «День».
- Неделя — 7 вертикальных колонок с событиями.
- 3 дня — то же, но на 3 дня от выбранной даты.
- День — список событий за сутки и кнопка «Добавить событие».
- Список (Agenda) — сгруппированные по датам события на 30 дней вперёд.

Переключение вида — через меню в AppBar. Стрелки листают соответствующий диапазон (месяц/неделя/3 дня/день/30 дней). После операций (создание/изменение/удаление) данные текущего вида перезагружаются автоматически.

Редактор события

Поля: Название (обязательно), Описание (опционально), Весь день, Начало/Окончание (локальные пикеры → сохраняются в UTC), Календарь (локальный), Повтор.

Повторы (RRULE, поддержка на MVP):
- Без повтора; Ежедневно; Еженедельно (в день недели начала); По рабочим дням (MO,TU,WE,TH,FR);
- Ежемесячно (по числу, BYMONTHDAY);
- Ежемесячно (по дню недели, позиционный BYDAY — например, 1MO, -1SU);
- Ежегодно (BYMONTH + BYMONTHDAY).
- По умолчанию у повторов установлено ограничение по времени UNTIL = +1 год (чтобы серии не «разливались» на весь календарь). Экспансия повторов выполняется только в окне текущего вида.

Редактирование/удаление повторяющихся событий (как в Google Calendar):
- Только это повторение — создаётся override‑инстанс (RECURRENCE‑ID), базовой серии добавляется EXDATE. При удалении — добавляется EXDATE; при удалении override — EXDATE удаляется.
- Это и все последующие — базовая серия «режется» (в RRULE проставляется UNTIL на момент выбранного инстанса −1 сек), все override на и после точки удаляются; далее создаётся новая серия с параметрами редактора, начиная с выбранного инстанса (при удалении — новая серия не создаётся).
- Вся серия — правка/удаление базовой записи.

Где хранятся данные и как их «почистить»
- JSON‑файл: ~/.potok_calendar.json.
- Можно удалить файл, чтобы сбросить календарь (приложение создаст дефолтный календарь «Личный»).

Тестовые сценарии (чек‑лист)

1) Создание одиночного события — месяц → долгое нажатие на дне → форма → Сохранить → проверить в месяце и в дне.
2) Ежедневный повтор — создать сегодня, убедиться, что инстансы видны в окне и серия ограничена по времени.
3) «Только это повторение» — изменить инстанс, убедиться: изменился только он.
4) «Это и все последующие» (правка) — изменить инстанс, убедиться: до даты всё прежнее, начиная с даты — изменилось.
5) «Только это событие» (удаление) — удалить один инстанс; остальные остались.
6) «Это и все последующие» (удаление) — удалить хвост серии; до даты инстансы на месте.
7) «Вся серия» — удалить серию целиком; пропали все инстансы.
8) Месячное «по дню недели» — проверить позиционный BYDAY (1‑й понедельник).
9) Хвостовые дни — событие на границе месяцев видно в сетке соседнего месяца.

Календарный REST API (заглушки)

- Спецификация: openapi/calendar.yaml (OpenAPI 3.1).
- Бэкенд‑роутер (chi) смонтирован под /v1/* вместе с middleware (request‑id, logging, auth‑заглушка).
- Доступные эндпоинты (минимальные ответы 200/201/204):
  - GET /healthz, GET /readyz
  - GET/POST /v1/calendars, GET/PATCH/DELETE /v1/calendars/{uid}
  - GET/POST /v1/events, GET/PATCH/DELETE /v1/events/{uid}
  - GET/PUT/POST/DELETE /v1/events/{eventUid}/reminders[...]
  - GET /v1/sync/delta
  - POST /v1/import/ics, GET /v1/export/ics
- Примечание: фронтенд пока не вызывает эти маршруты — хранилище офлайн.

Траблшутинг

- Не обновляется экран «День» после возврата — исправлено (экрана stateful, авто‑перезагрузка). Обновите сборку.
- Красные полосы OVERFLOWED в месяце — высота ячеек увеличена. Если повторяется, сообщите.
- Ошибка URI для заметок — NoteService использует http://localhost:8080. Для Android‑эмулятора используйте http://10.0.2.2:8080.
- Очистить календарные данные — удалите ~/.potok_calendar.json.

План развития (кратко)

- Drift + Riverpod + FTS5 для календаря, миграции и поиск.
- Полная поддержка RRULE (сложные BYDAY, WKST, COUNT UI), EXDATE‑редактор, overrides.
- Drag‑create/resize, раскладка перекрытий, линия «текущее время».
- Нотификации и часовые пояса.
- Генерация dart‑dio клиента по openapi/calendar.yaml и SyncProvider.

Модуль «Доски» (Kanban, локальный API)

- Где в UI: нижняя вкладка «Доски». Список досок → экран доски.
- Базовые возможности: колонки (WIP‑лимит), карточки, перетаскивание между колонками, поиск по тексту, карточка с чек‑листом/комментариями/тегами.
- REST (локально под /v1/*):
  - Boards: GET/POST /v1/boards
  - Columns: GET/POST /v1/boards/{boardId}/columns, PATCH/DELETE /v1/columns/{id}, PUT /v1/boards/{boardId}/columns/order
  - Cards: GET/POST /v1/boards/{boardId}/issues?column_id=&search=&tags=, GET/PATCH/DELETE /v1/issues/{id}, POST /v1/issues/{id}:move
  - Checklists: GET/POST /v1/cards/{id}/checklist, PATCH/DELETE /v1/checklist_items/{id}
  - Comments: GET/POST /v1/cards/{id}/comments, DELETE /v1/comments/{id}
  - Tags: POST /v1/cards/{id}/tags:bulk, DELETE /v1/cards/{id}/tags/{tag}
- Примечания:
  - Перемещение учитывает WIP‑лимит колонки (ошибка 409). В UI показывается Snackbar «WIP‑лимит колонки достигнут».
  - Позиция карточки внутри колонки задаётся целым числом; при переносе ставится в конец.
  - Due Date карточки создаёт событие в календаре (UTC, 1 час, «Due: …»).
