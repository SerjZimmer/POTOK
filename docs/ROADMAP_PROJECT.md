# Дорожная карта проекта POTOK (общая)

Цель: собрать «сквозное» видение развития приложения (Заметки + Календарь + инфраструктура) с учётом перехода от офлайн‑первого MVP к онлайн‑сервису, рассчитанному на 200 000 пользователей (календарь и почта/уведомления).

## 0. Архитектурные принципы

- Offline‑first: клиент должен быть полезным без сети; синк — дельтами, идемпотентно, с конфликт‑политиками.
- Простые контракты: OpenAPI — источник правды; версионирование `/v1`.
- Наблюдаемость и надёжность: метрики, логирование, трассировка, алерты — обязательны в «онлайне».
- Масштабируемость: stateless API, очереди/кэш, БД с партиционированием и репликами.
- Безопасность: TLS, OAuth2/JWT, RBAC, rate‑limits, защита персональных данных.

## 1. Модули

### 1.1 Календарь — см. `docs/ROADMAP_CALENDAR.md`

Ключевые вехи: Drift/Riverpod/FTS5 → REST‑синк → iTIP/iMIP/FreeBusy → CalDAV → масштаб.

### 1.2 Заметки — см. `docs/ROADMAP_NOTES.md`

Ключевые вехи: поиск/UX → OpenAPI/ETag → PostgreSQL → синк → масштаб.

### 1.3 Почта/уведомления (дорожная карта высокого уровня)

- MVP: каналы уведомлений (push/email) для календаря (iMIP), сервис почтовой отправки (SMTP relay) + Webhook для статуса доставки.
- V2: приём приглашений по email, сопоставление с пользователями (iMIP).
- V3: собственный почтовый шлюз (при необходимости):
  - SMTP входящий/исходящий, хранение писем (объектное хранилище + индексы в БД).
  - IMAP/POP3 (опционально) или только «почтовые события» для интеграции.
  - Антиспам/DKIM/SPF/DMARC — интеграция с существующими провайдерами.
- Масштаб: очередь задач (NATS/Kafka), «почтовые» воркеры, идемпотентность, retry‑политики.

## 2. Платформа/инфраструктура

- Среды: dev, staging, prod; IaC (Terraform) + GitOps (ArgoCD/Flux) — по мере роста.
- Контейнеризация: Docker + Kubernetes (минимум k8s на prod), сервис‑мэш (по необходимости).
- Секреты: Vault/Sealed Secrets.
- Логи/метрики: Loki/ELK, Prometheus + Grafana, OpenTelemetry.

## 3. Безопасность и комплаенс

- Хранение персональных данных — шифрование at‑rest (PostgreSQL TDE/инфраструктурные средства), TLS everywhere.
- Аутентификация: OAuth2/OpenID Connect; MFA (по требованию).
- Авторизация: per‑tenant доступ, ACL на календари/папки/заметки.
- Аудит‑лог изменений (особенно для календаря и рассылок приглашений).

## 4. Производительность (целевые метрики)

- Листинг календаря/заметок: p95 < 200 мс на диапазон/папку, p99 < 500 мс.
- Синк дельт: p95 < 300 мс, частота фонового синка — от 30 сек до 5 мин.
- Free/Busy многопользовательский запрос (до 50 пользователей): p95 < 400 мс.
- Отправка iMIP‑приглашения: < 2 сек до постановки в очередь; SLA доставки — зависит от почтового провайдера.

## 5. План релизов (интегрально)

1) Стабильный офлайн (календарь/заметки), UX‑улучшения, FTS5, Drift/Riverpod.
2) Сервисный слой: OpenAPI, chi, ETag, аутентификация, базовая телеметрия.
3) PostgreSQL, миграции, поиск, очереди, кэш, инфраструктурные окружения.
4) Синхронизация delta/anchors, фоновые задачи.
5) iTIP/iMIP + Free/Busy; уведомления (email/push).
6) CalDAV/CardDAV; интеграции с внешними клиентами; масштабирование под 200k.

## 6. Риски

- Сложность календарных правил и конфликтов синка — нужны интеграционные тесты и тестовые наборы из реальных данных.
- Нагрузочные пики (массовые приглашения/напоминания) — очереди, лимиты, backpressure.
- Кроссплатформенные нюансы TZ и all‑day — тесты на переходах и края диапазонов.
